# CAS 配置页开发规范

这个目录下的 CAS 配置页已经按**页面编排层 + 功能卡片 + composable 动作逻辑**拆分。

## 目录结构

- `index.vue`
  - 页面编排层
  - 只放跨卡片共享状态、计算属性、联动逻辑
- `components/*Card.vue`
  - 每个功能区一个组件
- `composables/useXxx.ts`
  - 带确认弹窗 / 异步执行 / loading / message 的动作逻辑

## 后续开发必须遵守

### 允许做的
- 给某个功能区新增字段、展示、局部交互
- 为某类动作新增 composable
- 当卡片继续变复杂时，再拆更小组件

### 禁止做的
- 把新按钮、新弹窗、新接口调用直接堆回 `index.vue`
- 把多个功能区重新揉成一个超长单文件组件
- 用补丁式开发把重复 script/style/template 残片塞进页面
- 为了省事留下“大泥坑单文件”技术债

## 新增需求时怎么判断放哪里

1. **卡片私有逻辑** → 放卡片组件里
2. **可复用动作逻辑** → 放 `composables/useXxx.ts`
3. **跨卡片共享状态** → 才放 `index.vue`
4. **卡片过厚** → 继续拆，不要硬撑

## 交付前检查

- `npm run build` 必须通过
- 不要有未使用 import / 未使用状态
- 不要有重复 `<script>` / `<style>` / 脏尾巴
- 页面结构不能回退成单文件大泥坑

## 当前业务边界（已更新）

- 不再走“云端 `.cas` 直转 + 缓存表”方案
- 订阅阶段只负责把 `.cas` 下载到本地 `/local_cas/...`
- 恢复阶段只使用本地 `.cas` 文件
- 旧的缓存表按钮/接口/说明都应逐步删除，不要再新增类似能力

## 给 AI 的直接约束

> 修改 CAS 配置页时，默认先拆组件/抽 composable，不要为了快把逻辑重新堆回 `index.vue`。
